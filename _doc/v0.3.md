# git4 - 開發紀錄 v0.3

**日期**: 2026-05-01
**版本**: v0.3

## 開發摘要
根據專案藍圖 (AGENTS.md) 所規劃，v0.3 的重點在於「暫存區管理的升格」與「狀態比對指令 (`status`)」。在此版本中，我們實作了完整且遞迴的比對引擎，使得 `git4` 能夠區分出檔案處於 `HEAD`、`Index` (暫存區) 與實體工作目錄三者之間的不同步狀態。

## 實作細節

### 1. `git4 status` 狀態比對引擎
- 新增了 `status` 功能，內部實作了三層狀態的資料庫收集：
  1. **獲取 HEAD 狀態**: 透過解析當前 HEAD Commit 中的 Tree 指標，實作了 `read_tree_recursive` 遞迴將所有子資料夾下的 Blob 展平成 `{ 絕對路徑: Hash }` 字典。
  2. **讀取 Index**: 直接匯入本機隱藏檔 `.git4/index` 內的索引快取。
  3. **掃描 Workspace**: 針對目前實體的資料夾遞迴計算 Hash (`get_workspace_files`)。
- **三向多維度比對**:
  - **Changes to be committed**: 比對 `Index` 鍵值與 `HEAD` 的差異。如果在 Index 有更新的 Hash 而 HEAD 裡沒有或不同，視為新增或已修改備妥的檔案。
  - **Changes not staged for commit**: 比對 `Workspace` 和 (Index/HEAD聯集) 之間的差異。如果在磁碟上的實體 Hash 跟歷史追蹤版本不同，則為尚待 `add` 的項目。若被追蹤但被磁碟刪除，則會標示 `deleted`。
  - **Untracked files**: 完全沒有出現在 HEAD 或 Index 裡面的新檔案。

### 2. 相容與整合
- 各項底層如取得 Tree Hash、忽略規則 (.git4、target、git.sh) 等皆進行了重構，使狀態分析更具精準度。

### 3. 自動化測試腳本升級
- 再次擴編了 `test.sh` 加入 `[8/8] 測試 status 功能`。
- 在腳本中，驗證了加入新檔案、被修改但不 `add` 等狀態下，`status` 所吐出的報表都能精確指出未暫存 (not staged) 與將要提交 (to be committed) 這兩種不同階段的狀態變化。

## 備註與下一步 (未來看法)
- 目有目前 `git4 checkout` 並沒有將 Index 暫存區連帶清空並同步成新分支的狀態。這導致在測試腳本中，若我們切換分支，原本暫存區內的紀錄可能會被一併帶過去。未來我們需要改寫 `checkout` 以及 `commit` 流程，使一切圍繞著受保護的 `index` 運轉，以達成 100% 擬真。
- v0.4 預計實作檔案內容差異顯示 (`diff`) 甚至是基礎合併機制。
