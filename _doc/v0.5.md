# git4 - 開發紀錄 v0.5

**日期**: 2026-05-01
**版本**: v0.5

## 開發摘要
在 v0.5 當中，我們完成了遠端資料與打包操作的探討，使 `git4` 具備了跨資料夾（模擬跨伺服器）的資料對拷與傳輸能力。這補足了建立分散式版控系統最關鍵的「協同合作」與「遠端進度同步」的這塊拼圖。

## 實作細節

### 1. `git4 clone`：遠端倉儲對拷
- 實作了跨資料夾 `.git4` 基礎目錄的遞迴複製 (`copy_dir_recursive`)，能確保在執行 `git4 clone <source> <dest>` 時，來源目錄的所有歷史物件皆被完整且獨立地備份至目的地。
- 對拷結束後系統會自動分析目的地的最新分支 (`HEAD`) 並呼叫內建的 `checkout` 方法將實體檔案完美還原，這意味著 `clone` 之後立刻就可以進行開發。

### 2. `git4 push`：主動式資料推送
- 當執行 `git4 push <remote_path> <branch>` 時，本地 `git4` 會將本身專案內的 `objects` 目錄所有的 Blob / Tree / Commit 全面與遠端同步合併。
- 完成傳輸後，對遠端的 `refs/heads/<branch>` 分支指標進行強制指向本地最新的 Commit Hash，以此達成進度推播的目的。

### 3. `git4 fetch`：拉取遠端最新進度
- 指令 `git4 fetch <remote_path>` 提供了反向邏輯：不但把遠端倉庫的 `objects` 同步回本機資料夾，還會在本地目錄下的 `refs/remotes/origin/` 開闢並記錄遠端的分支狀態。
- 這樣的好處是本地端就能解析遠端的最新指標，日後便能支援如 `git4 merge remotes/origin/main` 這種追蹤合併。

### 4. 端對端自動測試擴增
- 於 `test.sh` 中補全了第 10 階段的聯網對拷實測：
  1. 將開發中的 `test_repo` 透過 `clone` 拷貝產生全新的 `remote_repo`。
  2. 在 `remote_repo` 中進行獨立改寫與 `commit`，並透過 `push` 送回原本的 `test_repo`。
  3. 切回 `test_repo` 時，輸入 `log` 成功確認讀取到了由另一端發送過來的跨目錄提交歷史！全數完美通過。

## 結語與未來展望
至此，`git4` 這個基於 Rust 開發的極簡地端系統已經完整經歷了創立、暫存、遞迴樹封裝、Hash壓縮、多線分支管理、比對甚至到跨資料夾傳輸推送的核心功能。未來若有需要更深入的改良，例如建置真正的 TCP / HTTP Server 接受通訊、實作 Myers 三方合併等，都可以基於此 MVP （包含自動測試涵蓋）繼續穩定地疊代。
