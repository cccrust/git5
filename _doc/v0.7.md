# git4 - 開發紀錄 v0.7

**日期**: 2026-05-01
**版本**: v0.7

## 開發摘要
v0.7 的任務是跨越 Git 生態當中最具決定性但也最複雜的一環 —— **Packfile 機制**。
我們成功手動實現了一個原生的 `.pack` 解壓與物件打散引擎 `git4 unpack-objects`，讓專案真正具備了接收與相容 GitHub 真實通訊包裹的能力。

## 實作細節

### 1. 二進位 Pack Header 與 VLE 解析
- **辨識包裹**: 實作程式讀取前 12 位元組，準確驗證 `PACK` 魔術特徵碼，並確保版本為 `2`，接著算出包裹內部總共蘊含了幾顆物件。
- **Variable-Length Encoding (變長度編碼) 解析能力**: 負責從無情的二進位陣列中擷取出物件的屬性與長度。依靠位元平移（Shift `<< 7`）來組裝並還原原始尺寸。對於 `OFS_DELTA` 特殊的 128 進制基數偏移量 (Negative Offset) 的演算法也在這項工程中被成功再現。

### 2. 資料串流萃取引擎 (`ZlibDecoder`)
- **解壓縮邊界挑戰**: 傳統的解壓縮通常是整包解開，但在 `.pack` 裡面每一個壓縮物件是緊密接續在一起的。為了避免 Over-read 吃掉下一個物件的標頭，我們改用最底層的 `flate2::Decompress`。
- 這個方法能逐次解析出純淨的位元元組，並且透過 `total_in()` 回報我們已經在原串流中消耗了多少體積，使得游標 (pos) 能夠完美停在下一個物件的開端！

### 3. 解離為鬆散物件 (Loose Objects)
- 當物件成功解壓縮成原始資料後，程式會自動依照它的屬性重新組合出 Git 最傳統的 `"{type} {size}\0{content}"` 標準格式。
- 再度投入 `sha1` 重新計算出這包資料的 40 字元真身 Hash，最終利用我們自己的 `write_object` / Zlib 機制，華麗地寫進 `.git4/objects` 裡，變成隨時可以 `checkout` 出來的常規物件！

### 4. 與原廠 Git 直接碰撞的自動化測試
- 在 `test.sh` 中加入了第 12 塊測試，這是我個人最滿意的一環。
- 我在這個測試中調用了真正的正版 `git repack -a -d` 指令產生一個壓縮檔 `.pack`。緊接著呼叫我們剛誕生的主角：`git4 unpack-objects` 去硬生生啃掉正版 git 製造的壓縮球，結果我們成功吐出了 3 個正確的 Loose Objects！完全通過相容性驗證。

## 備註與下一步
- 目前程式能完全應付實體的 `Commit`, `Tree`, `Blob` 資料解離。但若遇到 `REF_DELTA` 或 `OFS_DELTA` (也就是差值補丁)，我們現在雖然能順利算出它們的體積並跨過它們（不影響整體迴圈），但還未做出 Apply Delta。不過對於我們即將引發的初次下載功能已經非常強大！
- v0.8，我們會迎擊 HTTP Packfile Download，結合 v0.6 與 v0.7！
