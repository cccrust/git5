# git4 - 開發紀錄 v0.1

**日期**: 2026-05-01
**版本**: v0.1

## 開發摘要
完成了 `git4` 的最小可行性架構（MVP），這個版本具備了基本的物件管理能力與基礎的地端版控機制。我們確立了專案結構以及與真實 `git` 物件儲存相容的設計，並設計了自動化測試確保核心流程的穩定運作。

## 實作細節

### 1. 核心物件儲存設計
- 實作了基於 `flate2` 的 Zlib 壓縮，與 `sha1` 雜湊計算，並以傳統 git 格式 `[物件類型] [大小]\0[內容]` 將物件儲存在對應的 `.git4/objects/[2字元]/[38字元]` 結構中。
- 支援了 **Blob** (儲存檔案)、**Tree** (儲存資料夾結構與權限) 與 **Commit** (儲存訊息與時間戳記) 等三大儲存物件。

### 2. 底層指令開發 (Plumbing)
- `hash-object`: 能針對特定的檔案轉化為 blob 並產生 hash (加上 `-w` 則寫入實際儲存庫)。
- `cat-file`: 利用 `-p` 解析給定的物件 hash，解開 zlib 壓縮並顯示出物件儲存的原始內容。
- `write-tree` 與 `commit-tree`: 能處理當前的目錄狀態並遞迴式地產生 tree 節點實體，最終串接產出 commit 物件。

### 3. 高階指令開發 (Porcelain)
- 開發了以 `clap` 為主幹的使用者介面。
- `init`: 初始建立工作環境。
- `add`: 實作了基礎的 index 機制，將加入的檔案更新於 `.git4/index` 並且建立 blob 物件。
- `commit`: 當呼叫時，系統自動捕捉所有資料並做為一顆 Tree，連同取得現行的 HEAD 更新分支參考。
- `log`: 實作了追蹤 `HEAD` 至各個 parent commit 並輸出歷史訊息的能力。

### 4. 品質確保
- 完成了 `test.sh` 端對端自動測試腳本，能在獨立 `test_repo` 環境中進行順暢的 `init` -> `add` -> `commit` -> 修改後加入第二板 -> `log` 的檢驗並驗證完全通過。

## 待優化 (未來展望)
- 目前 `commit` 時自動捕捉的機制雖然方便，但尚未完全比對與尊重 `index` 暫存區的選擇性。 
- 對於不同分支 (branches) 與指標的切換 (`git4 checkout` / `git4 branch`) 尚無實體化指令可以操作。
- 目錄下如果有過深或者忽略（如 `.gitignore` 的對讀取）邏輯可以再強化。
