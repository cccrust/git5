# git4 - 開發紀錄 v0.4

**日期**: 2026-05-01
**版本**: v0.4

## 開發摘要
在 v0.4 當中，我們完成了真正的專案版本比對演算法整合與合併機制。這標誌著我們從紀錄、切換，演進到了「分支協作」的地步。此次擴增了 `git4 diff` 與 `git4 merge`。

## 實作細節

### 1. 以 `similar` crate 結合字串比對的 `diff` 命令
- 引入了 Rust 輕量且功能強大的 `similar` 套件作為差異對比引擎。
- `git4 diff <file>` 功能原理：
  - 提取暫存區內（或 HEAD 落後紀錄）的檔案原始資料並進行解壓縮為字串。
  - 將本機最新的實體檔案讀取出來。
  - 利用 Myers 演算法 (`similar::TextDiff::from_lines`) 進行兩者逐行尋訪，並且輸出帶有 `+` 與 `-` 標記的典型差異排版，方便開發者直接理解變動。

### 2. Fast-forward 分支合併 (`merge`)
- 實作了 `git4 merge <branch>` 邏輯，目前支援最純粹的 Fast-Forward：
  1. 對目標分支執行 `resolve_revision`，並驗證合併安全性。
  2. 開發了內部通聯函式 `is_ancestor`：其追蹤了兩個 commit 的 `parent` 樹鏈。驗證當下 `HEAD` 是否確為目標 Commit 的直接祖先。
  3. 若確定可以直接追上 (Fast-forward)，則：
     - 利用既有 `restore_tree` 把目標樹狀檔案更新於工作目錄中。
     - 把 `HEAD` (主分支指標) 大步跨進至目標 Commit Hash。

### 3. 自動化測試驗證
- 在 `test.sh` 裡補全了階段 9 (`[9/9]`)。
- 測試情境包括：建立一條專門觸發合併測試的分支 (`feat-merge`)、做額外新增的文字變更並觸發 `diff`、隨後切換回主線 `main` 分支進行 `merge`，並印出合併後的 log。
- `./test.sh` 毫無報錯地通過了 Fast-forward 的檢核和記錄比對。

## 備註與下一步
- 目前合併僅涵蓋了簡單地追上祖先節點的 Fast-Forward 動作，真正在歷史分岔時的三方合併 (Three-way merge) 因須解衝突等高複雜度，將保留到或許將來的版本探討。
- 下一步 (v0.5) 則可依計畫嘗試加入跨遠端目錄交換資料的封裝機制 (`clone`, `fetch`, `push` 概念原型)。
